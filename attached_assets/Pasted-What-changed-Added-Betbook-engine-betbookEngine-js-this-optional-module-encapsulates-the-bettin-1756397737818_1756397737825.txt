What changed

Added Betbook engine (betbookEngine.js) – this optional module encapsulates the betting context. It is only invoked when the user swipes left on an alert; the primary alert payload remains free of any wagering advice. The engine returns stubbed odds, a generic AI advisory statement reminding the user to do their own research, and example sportsbook links. When APIs are available, you can replace the stubbed data with real odds and AI output.

Updated user settings (userSettings.js) – introduced a isBetbookEnabled() helper which checks an optional betbookEnabled flag. This allows users to opt in or out of seeing Betbook content independently of their normal alert tiers.

CHANGELOG updated (CHANGELOG.md) – documented the new Betbook engine and clarified that betting information is never included in the primary alert; it only appears in the secondary view when triggered by the user.

CHANGELOG.md:
[2025‑08‑28] - Mandatory alert-engine laws for ChirpBot v3
Added

Game status gating – The alert engine now strictly respects the game’s lifecycle.

Only games marked Live are actively monitored.

Scheduled games are ignored until they go live.

Final games are archived and removed from monitoring.

Multi‑level alert model – Introduced a four‑tier alert structure to surface high‑leverage game situations while preventing spam:

Level 1 – Fail‑safe hard logic: Always runs when a game is live. Searches for pre‑defined high‑probability scoring conditions (e.g. RISP, bases loaded, clutch time). Ensures alerts continue even if higher‑level modules fail.

Level 2 – Player & historical triggers: Activates when player stats or trends amplify scoring odds (e.g. power hitter on deck with runners on base).

Level 3 – Weather & environmental factors: Triggers when wind, temperature or other conditions materially affect scoring (e.g. wind blowing out ≥ 10 mph).

Level 4 – AI synthesis: A final layer that synthesizes all data (scores, player trends, weather) to generate natural‑language alerts. Can produce alerts even when Level 1 is false, provided the AI is very confident.

Tier decision rules – Alert tier is determined by explicit IF chains:

If L1 = yes, L2 = no, L3 = no → emit Alert 1.

If L1 = yes and exactly one of L2 or L3 = yes → emit Alert 2.

If L1 = yes, L2 = yes, and L3 = yes → emit Alert 3.

If L1 = no, but the AI is highly confident → emit Alert 4 (AI‑only path).

Deduplication & cooldown – Added context‑aware keys (sport, game ID, inning/period, outs/time, runners, batter) to avoid duplicate alerts. Each alert level defines a cool‑down window to prevent repeated messages about the same state.

Supersession of alert levels – When multiple alert levels could trigger for the same game context, only the highest‑level alert is emitted. Lower‑level alerts are suppressed if a higher‑level trigger is satisfied either in the same evaluation or shortly thereafter. For example, if both L1 and L2 are true, the system sends Alert 2 and does not send Alert 1. If an L2 condition arises after an L1 alert has already been emitted for the same situation, the L2 alert supersedes the earlier L1 alert; only the L2 alert is delivered.

User settings override – The alert engine now honours per-user preferences. Before emitting any alert, the system checks the user’s notification settings. A global toggle can disable all alerts, and users can enable/disable specific tiers (Level 1–4) or sports. Alerts are only sent if the user’s settings allow them. This ensures that if a user has alerts turned on in the settings panel, those alerts will still be delivered regardless of the new four‑rule law system; conversely, if they are disabled, no alerts are sent.

Notes

These rules are mandatory and form the baseline for ChirpBot v3’s alert engine.
Developers must ensure that all sport‑specific engines adhere to these gating and tiering rules.
Implementations should hook into these levels via clearly defined functions (runLevel1HardCoded, runLevel2PlayerHistory, runLevel3Weather, runLevel4AISynthesis) and respect the master status switch before any alert logic runs.

Added

Betbook engine (experimental) – Introduced an optional module named the Betbook engine. The core four‑tier alert system continues to surface only game‑related situations; it never delivers gambling or wagering advice. When a user performs a secondary action on an alert (such as swiping the alert card to the left), the Betbook engine may be invoked to present supplemental sports betting context, AI‑generated insights, and links to partner sportsbooks. The engine is disabled by default unless the user enables it via the new betbookEnabled setting. Developers can integrate the Betbook engine by calling getBetbookData(alertContext) from the UI layer; the function returns odds (stubbed in this demo), a generic AI advisory statement reminding users to conduct their own research, and example sportsbook URLs. To maintain compliance, do not embed betting content directly in the primary alert payload, and ensure user preferences are respected when showing Betbook data.

Added

MLB scoring probability model – Introduced a math‑driven module (mlbAlertModel.ts) that estimates the probability of at least one run scoring in the remainder of an inning across all 24 base/out states. This model applies context modifiers (leverage, power hitters, pitcher risk, weather and park factors) and classifies the adjusted probability into Low, Medium, or High severity bands. Developers can wire the associated helper functions (mlbL1WithProb, mlbL2WithProb, mlbL3WithProb) into their existing sport logic engines to align the Layered Alert levels (L1–L3) with estimated scoring odds: Low (≥ 65%), Medium (≥ 70%), High (≥ 80%).

Low (≥ 65% and < 70%) → qualifies the situation for Level 1; recommended priority hint ~75.

Medium (≥ 70% and < 80%) → qualifies for Level 2; recommended priority hint ~85.

High (≥ 80%) → qualifies for Level 3; recommended priority hint ~95.

The model returns an explanatory reason array that lists each context adjustment applied. These reasons should be appended to the alert’s justification block for auditability.
