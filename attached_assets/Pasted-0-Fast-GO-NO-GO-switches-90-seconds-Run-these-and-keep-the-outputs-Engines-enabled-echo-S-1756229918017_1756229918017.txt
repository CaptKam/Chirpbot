2) Source → “Live” detection

Every source must normalize to the same predicate:

export function isLiveStatus(s: any): boolean {
  const abs = (s?.abstractGameState ?? "").trim();
  const coded = (s?.codedGameState ?? "").trim();
  const text = (s?.detailedState ?? s?.status ?? "").toLowerCase();
  return abs === "Live" || coded === "I" || text === "live" || text.includes("in progress");
}


Smoke check:
If /api/tennis/matches or /api/mlb/games is empty while events are live → your status mapping is off. Fix normalization in each source before anything else.

5) Alert attempts vs. emits (prove the core is running)

Log these in every engine:

ALERT_ATTEMPT { sport, type, prio, key }

DEDUP_BLOCK { sport, type, key, reason }

ALERT_EMIT { sport, type, id, prio }

METRICS_<SPORT> { live, statesSeen, attempts, emitted, suppressed, drops } (every 60s)

Interpretation:

Attempts = 0 → conditions never true or settings gating too strict.

Attempts ≫ Emitted with many DEDUP_BLOCK → key/window wrong (see §6).

Emitted > 0 but no UI → delivery problem (see §7).

6) Dedup: stop dupes without killing signals
Tennis: point identity + stage edge

Use a stable point key so repeated polls don’t duplicate:

function stablePointKey(s) {
  const set = s.currentSet, g = `${s.gamesInSet.home}-${s.gamesInSet.away}`;
  const tb = s.isTiebreak ? `TB:${s.tbPoints.home}-${s.tbPoints.away}` :
                             `P:${s.points.home}-${s.points.away}`;
  const sv = s.server ?? "-";
  return `S${set}:G${g}:${tb}:SV${sv}`;
}


Emit only on rising edge of the stage (BP → DBP → SP → MP). Keep escalationOnly=true and 15s windows for point-scoped tennis alerts.

DB idempotency (safety net)

Add a unique hash so even races can’t dup:

alter table alerts add column if not exists dedup_hash varchar(64);
create unique index if not exists ux_alerts_dedup on alerts(dedup_hash);


Hash the dedup key before insert and ON CONFLICT DO NOTHING.
