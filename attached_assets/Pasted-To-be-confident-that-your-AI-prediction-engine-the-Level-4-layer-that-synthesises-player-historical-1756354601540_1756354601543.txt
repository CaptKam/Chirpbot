To be confident that your AI prediction engine (the Level 4 layer that synthesises player/historical data, weather, and game context) is running correctly, you need to treat it like any other microservice: expose its health and readiness, monitor its resource consumption, and build fall‑back behaviour when it’s unhealthy. Here’s a practical approach:

Expose health‑check endpoints. Add a lightweight HTTP route (e.g. /healthz) that returns OK when the model server is alive and responsive. Deployers using Kubernetes should configure a liveness probe against this endpoint so the orchestrator can restart the engine if it stops making progress. Implement fail‑over behaviour. If the health or readiness probe fails, or if your scheduled checks report sustained errors, the engine should enter a degraded mode where Level 4 alerts are disabled. Your alert coordinator already includes a fail‑safe: Level 1–3 engines should run regardless, so you still deliver baseline alerts even when AI is unavailable. Log the degraded state so you can track outages. When the AI service becomes healthy again, automatically re‑enable it.

Verify via user settings. Respect per‑user notification preferences before sending AI‑driven alerts. If the user has turned off AI predictions in the settings panel, skip the Level 4 call. Conversely, if they have AI alerts enabled, your coordinator should always call the AI engine once the health and readiness checks succeed.
