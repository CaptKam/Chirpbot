Your normalizer might not mark status:'LIVE' (common). Make mapStatus tolerant:

function mapStatus(r:any): 'SCHEDULED'|'LIVE'|'FINAL'|'DELAYED' {
  const s = String(r.status ?? r.abstractGameState ?? '').toLowerCase();
  if (/final|ended|complete/.test(s)) return 'FINAL';
  if (/live|in\s*progress|top|bottom|1st|2nd|3rd|4th|ot|q\d/.test(s)) return 'LIVE';
  if (/delay/.test(s)) return 'DELAYED';
  return 'SCHEDULED';
}

5) One-file “status snapshot” (paste & hit)

Add a quick route to tell you what’s happening right now:

// server/http/status.ts
import express from 'express';
import { Pool } from 'pg';
export const status = express.Router();
const pool = new Pool({ connectionString: process.env.DATABASE_URL });

status.get('/admin/status', async (_req, res) => {
  const { rows } = await pool.query(`
    SELECT
      count(*) FILTER (WHERE created_at >= now()-interval '5 minutes') AS alerts_5m,
      count(*) FILTER (WHERE state='DELIVERED' AND created_at >= now()-interval '5 minutes') AS delivered_5m
    FROM alerts;
  `);
  res.json({
    ok: true,
    maintenance: process.env.MAINTENANCE_MODE === '1',
    ruleset: process.env.RULESET_ACTIVE,
    stats: rows[0]
  });
});

Most common root causes (and the quick fixes)

Maintenance still on → MAINTENANCE_MODE=0, restart.

Status mapping too strict → use the tolerant mapStatus above.

WS URL wrong / mixed content → use location.origin.replace(/^http/,'ws').

No WS broadcast in notifier → call broadcastAlert when you pushAlert.

Cooldowns too long while testing → shorten values in config/cooldowns.json.

Dedup key too broad (e.g., missing phase/situation) → ensure alert_key uses sport|gameId|phase|type|situation|....
