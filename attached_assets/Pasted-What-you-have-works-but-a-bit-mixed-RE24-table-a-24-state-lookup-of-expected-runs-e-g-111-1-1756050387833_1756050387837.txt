What you have (works, but a bit mixed)

RE24 table → a 24-state lookup of expected runs (e.g., 111-1 = 1.54).

Keying → first second third - outs (e.g., 101-1).

Probability proxy → prob ≈ RE / 2.2, clamped to 5–98%, then late-inning + AI + weather multipliers.

Priority → derived from final probability, with additive boosts (bases loaded, late/close, weather).

Pipes → dedupe, cache, Telegram/UI.

This is fast and understandable—but it conflates two different signals:

Expected runs (RE) ≠ probability of ≥1 run (RP). RE/2.2 is a heuristic that can misrank states (outs matter a lot).

One correction you should make

Don’t treat comments like 111-2 = 0.76 // 76% chance as true. 0.76 is expected runs, not 76% probability.

Either (a) lookup RP directly from data (recommended), or (b) learn a small mapping from (RE, outs, bases) → P(run≥1).

Minimal upgrade (drop-in, keeps your shape)

Add RP24 alongside RE24
Store both per state: { RE, RP }. Use RP for “will a run score?” alerts; use RE for “how big could the inning be?” context.

Pick one place for leverage
EITHER multiply the probability by a leverage factor OR add leverage in priority—not both (avoid double counting).

Smooth priority from prob
Replace additive boosts with a smooth mapping (logistic) from probability to 0–100. Keeps rankings honest and stable.

Calibrate
Track reliability (predicted vs. actual). If biased, apply isotonic/Platt calibration so 70% means ~70% over time.
