The way your RE24 system is wired explains why you’re seeing “a lot” of alerts. As implemented, getActiveRE24Level() always returns some RE24 level whenever the corresponding toggle is enabled, regardless of how likely a run actually is. Combined with the multiplicative leverage factors (15 % for late innings, 10 % for one‑run games, 8 % for runners in scoring position
) and the AI/weather multipliers, it’s very easy for finalProbability to saturate. In fact, the hybrid engine multiplies the base scoring probability by the leverage, AI, and weather factors and then clamps the result between 2 % and 98 %
; anything in a late inning with runners on can jump to ~0.98, which the logistic priority function then turns into a max‑priority alert
. Because getActiveRE24Level() doesn’t check this probability, every poll produces an alert, and when combined with bases‑loaded/RISP alerts you end up with a flood.

Recommendations to reduce alert spam

1. Gate RE24 alerts by probability and leverage.
Add a minimum probability threshold (and optionally require “high leverage”) before emitting any RE24 alert. For example, only send Level 3 when finalProbability ≥ 0.70, Level 2 when ≥ 0.50 and Level 1 when ≥ 0.30, otherwise return null so that no RE24 alert is created. You can compute the hybrid analysis once in getActiveRE24Level() and gate accordingly:

const MIN_PROB = { 1: 0.30, 2: 0.50, 3: 0.70 };

export async function getActiveRE24Level(
  gameState: MLBGameState, settings: any
): Promise<RE24LevelResult | null> {
  const alertTypes = settings?.alertTypes || {};
  // compute hybrid analysis once
  const hybrid = await analyzeHybridRE24(gameState);
  // Skip RE24 alerts if probability is low and it’s not high leverage
  if (hybrid.finalProbability < MIN_PROB[1] && !hybrid.isHighLeverage) {
    return null;
  }
  if (alertTypes.re24Level3 && hybrid.finalProbability >= MIN_PROB[3]) {
    const lvl3 = await calculateRE24Level3(gameState);
    return { ...lvl3, probability: hybrid.finalProbability };
  }
  if (alertTypes.re24Level2 && hybrid.finalProbability >= MIN_PROB[2]) {
    const lvl2 = await calculateRE24Level2(gameState);
    return { ...lvl2, probability: hybrid.finalProbability };
  }
  if ((alertTypes.re24Level1 || alertTypes.useRE24System)
      && hybrid.finalProbability >= MIN_PROB[1]) {
    const lvl1 = await calculateRE24Level1(gameState);
    return { ...lvl1, probability: hybrid.finalProbability };
  }
  return null;
}


With this change, ordinary base‑state updates no longer trigger RE24 alerts; only genuinely threatening innings (e.g. 60 %+ scoring probability or high leverage) will surface.

2. Adjust the leverage multipliers.
The current factors (1.15 for inning 8+, 1.10 for one‑run games, 1.08 for RISP) stack aggressively
. Consider lowering them to 1.05/1.03/1.03 or even replacing the multiplicative approach with additive bonuses (e.g. multiply by 1 + 0.05*(#conditions)). This prevents the finalProbability from rocketing to 0.98 in common situations
.

3. Tune the logistic priority mapping.
The logistic function currently treats any probability above 0.7 as nearly 100 % priority
. Raising the midpoint (m) to 0.8 or reducing the steepness (k) will reduce the priority for middling probabilities, helping the deduplication logic suppress lower‑value RE24 alerts.

4. Debounce RE24 alerts.
Even with gating, you may still see multiple RE24 alerts for similar states. After sending an RE24 alert, cache its re24Key (bases‑outs combination) and suppress further RE24 alerts for that key until either the base‑out state changes or some cooldown elapses. This complements your existing deduplication and keeps high leverage innings from spamming the same message.

5. Provide user‑configurable thresholds.
Expose the minimum RE24 probability as a setting (e.g. re24MinProbability) in the DB and front‑end. Users who enjoy more granular updates can lower it; others can raise it for a quieter feed.

These changes will align RE24 alerts with true high‑leverage moments and drastically reduce unnecessary notifications.
