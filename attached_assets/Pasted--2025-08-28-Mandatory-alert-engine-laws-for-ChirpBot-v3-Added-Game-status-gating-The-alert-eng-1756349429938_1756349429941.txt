[2025‑08‑28] - Mandatory alert-engine laws for ChirpBot v3
Added

Game status gating – The alert engine now strictly respects the game’s lifecycle.

Only games marked Live are actively monitored.

Scheduled games are ignored until they go live.

Final games are archived and removed from monitoring.

Multi‑level alert model – Introduced a four‑tier alert structure to surface high‑leverage game situations while preventing spam:

Level 1 – Fail‑safe hard logic: Always runs when a game is live. Searches for pre‑defined high‑probability scoring conditions (e.g. RISP, bases loaded, clutch time). Ensures alerts continue even if higher‑level modules fail.

Level 2 – Player & historical triggers: Activates when player stats or trends amplify scoring odds (e.g. power hitter on deck with runners on base).

Level 3 – Weather & environmental factors: Triggers when wind, temperature or other conditions materially affect scoring (e.g. wind blowing out ≥ 10 mph).

Level 4 – AI synthesis: A final layer that synthesizes all data (scores, player trends, weather) to generate natural‑language alerts. Can produce alerts even when Level 1 is false, provided the AI is very confident.

Tier decision rules – Alert tier is determined by explicit IF chains:

If L1 = yes, L2 = no, L3 = no → emit Alert 1.

If L1 = yes and exactly one of L2 or L3 = yes → emit Alert 2.

If L1 = yes, L2 = yes, and L3 = yes → emit Alert 3.

If L1 = no, but the AI is highly confident → emit Alert 4 (AI‑only path).

Deduplication & cooldown – Added context‑aware keys (sport, game ID, inning/period, outs/time, runners, batter) to avoid duplicate alerts. Each alert level defines a cool‑down window to prevent repeated messages about the same state.

Notes

These rules are mandatory and form the baseline for ChirpBot v3’s alert engine.
Developers must ensure that all sport‑specific engines adhere to these gating and tiering rules.
Implementations should hook into these levels via clearly defined functions (runLevel1HardCoded, runLevel2PlayerHistory, runLevel3Weather, runLevel4AISynthesis) and respect the master status switch before any alert logic runs.
