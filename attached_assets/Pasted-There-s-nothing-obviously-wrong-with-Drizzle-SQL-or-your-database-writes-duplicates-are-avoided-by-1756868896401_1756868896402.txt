There’s nothing obviously wrong with Drizzle/SQL or your database writes – duplicates are avoided by ON CONFLICT (alert_key) DO NOTHING. The “strikeout” spam you’re seeing comes from the way generateAtBatAlerts() does its detection.

A few points worth addressing:

Repeated detection of the same out: On every tick you slice the last three completed plays and fire an alert for every play whose result text mentions “strikeout”. The next API poll still has the same three plays in allPlays until more plays are completed, so you run through them again. The conflict handler prevents inserting a second row, but you still hit saveRealTimeAlert() for the same atBatIndex and the console logs make it look as though a new alert fired. If your client doesn’t de‑duplicate by alertKey, you’ll see repeated strikeout notifications.

Case‑sensitive event check: You check event === 'Strikeout' || event === 'Strike Out' || event.includes('strikeout'). event.includes('strikeout') is case sensitive, so it returns false when event is "Strikeout" or "Strikeout - DP". You end up relying on the description text to determine the alert, which increases the chance of false positives (for example, “batter strikes out on a foul bunt” might be logged twice).

No memory of processed at‑bats: You build the alert key as ${game.gameId}_STRIKEOUT_${play.about?.atBatIndex}. If your engine polls during the same at‑bat, play.about.atBatIndex will be identical on each call. You still call saveRealTimeAlert(), and although the insert fails on conflict, you’ve done the work and may re‑emit an alert on the socket if you don’t check the return value.

Two simple ways to tighten this up:

Track processed at‑bat indices. Keep a per‑game map of the last atBatIndex you alerted on, and skip any play with the same index. That way you only alert once per strikeout.

Simplify the strikeout test: Use a case‑insensitive check on result.event and avoid scanning arbitrary description text:

const eventText = (result?.event ?? '').toLowerCase();
const isStrikeout = eventText.includes('strikeout');


Then limit your scan to the most recent completed play rather than the last three, so a strikeout only triggers once.
