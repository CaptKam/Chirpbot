I’ve completely rewritten both the MLB alert model and the MLB engine to implement the four‑stage alert flow (L1 trigger, AI refinement, Betbook insights, and user delivery). The updated mlbAlertModel now calculates scoring probabilities based on base/out states, applies situational modifiers, and returns both severity and suggested priority for alerts; it also exposes tiered checks for LOW, MED, and HIGH‑probability situations.

The mlb‑engine has been overhauled to use this model. Instead of a fixed “scoring situation” function, it now maps the live game state into the format expected by the alert model, checks the computed severity, and—if an alert is warranted—goes through all four stages:

Stage 1 (L1 Trigger) – Calls the alert model (calcMLBScoringAlert) to determine if a scoring event probability is high enough to fire an alert.

Stage 2 (OpenAI Refinement) – Uses a single instance of OpenAiEngine to generate a concise, 120‑character game alert based on the situation context. A fallback description is supplied if the AI call fails.

Stage 3 (Betbook Insights) – Calls getBetbookData from betbook-engine.ts to fetch AI‑generated betting advice and insights, with a fallback stub if the call fails.

Stage 4 (Delivery) – Creates the alert record through storage.createAlert, labels it as an MLB alert, stores the betbook data, and broadcasts it to connected clients.

New helper methods (mapGameStateForAlertModel, buildSituationContext, and buildFallbackDescription) translate between the game state format and the alert model, provide natural‑language context for the AI, and ensure a fallback message if AI services are unavailable. The engine now instantiates an OpenAiEngine once per instance, imports the alert model via CommonJS, and imports Betbook functionality. Deduplication, alert emission, and game monitoring remain intact.
