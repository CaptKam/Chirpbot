1) OpenAI call: correctness, latency, and cost

Issues

You pass the entire dynamic prompt as system, then a generic user message. That can dilute the instructions/context.

No request timeout/abort: fetch can hang after your 3s race fires → socket leak.

Length check is chars, not tokens (2,000 chars ≠ 2,000 tokens).

No retry/backoff for 429/5xx.

Free-text output is harder to validate than JSON.

Fixes

Put stable guardrails in system, put context in user, and ask for JSON.

Add AbortController tied to your 3s timeout.

Add bounded retries with jitter for 429/500.

Validate token length earlier (approx w/ 4 chars/token or count words).

private async callOpenAI(payload: {
  system: string; // short, stable rules
  user: string;   // built by buildSportSpecificPrompt()
}): Promise<any | null> {
  const apiKey = process.env.OPENAI_API_KEY;
  if (!apiKey) { console.warn('⚠️ OPENAI_API_KEY missing'); return null; }

  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), 2800); // under 3s race

  try {
    const body = {
      model: "gpt-4o-mini",
      messages: [
        { role: "system", content: this.sanitizeInput(payload.system).slice(0, 1200) },
        { role: "user", content: this.sanitizeInput(payload.user).slice(0, 8000) }
      ],
      max_tokens: 180,
      temperature: 0.5,
      response_format: { type: "json_object" }
    };

    // simple bounded retry
    for (let attempt = 0; attempt < 2; attempt++) {
      const resp = await fetch("https://api.openai.com/v1/chat/completions", {
        method: "POST",
        headers: { "Content-Type": "application/json", "Authorization": `Bearer ${apiKey}` },
        body: JSON.stringify(body),
        signal: controller.signal
      });

      if (resp.ok) {
        const data = await resp.json();
        const text = data.choices?.[0]?.message?.content || "";
        const sanitized = this.sanitizeAIResponse(text);
        // parse + zod-validate JSON here
        return JSON.parse(sanitized);
      }

      // retry only on 429/5xx
      if (resp.status === 429 || (resp.status >= 500 && resp.status < 600)) {
        await new Promise(r => setTimeout(r, 200 + Math.random()*250));
        continue;
      }

      console.error('❌ OpenAI error:', await resp.text());
      return null;
    }
    return null;
  } catch (e:any) {
    if (e?.name === 'AbortError') console.warn('⏱️ OpenAI aborted at 2.8s');
    else console.error('❌ OpenAI call failed:', e?.message);
    return null;
  } finally {
    clearTimeout(timer);
  }
}

4) Cache key & TTL

Issues

Key = {sport, gameId, alertType, inning/quarter, outs, away, home} is too coarse (MLB) and too game-specific (low reuse across games).

Fixed 30s TTL isn’t tempo-aware.

Type mismatch: elsewhere you mention caching a structured UnifiedAIResponse, but callOpenAI returns a string → cached.response.enhancedMessage may be undefined.

Fixes

Unify type: always cache a structured object {enhancedMessage, context, bet, window}.

Richer key (per sport):

MLB: add count, runners signature (e.g., 1-3), scoreDiff bucket, wind bucket.

NFL: add down&distance, yardline bucket, scoreDiff, clock bucket, redZone flag.

Adaptive TTL: 5–10s in high-tempo (late innings/2-min), 20–40s normal, 60–120s low-tempo.

private generateCacheKey(ctx: CrossSportContext): string {
  const tempoClock = Math.floor((ctx.clockSec ?? 0)/15);
  const scoreBucket = Math.min(5, Math.floor(Math.abs((ctx.homeScore-ctx.awayScore))/2));
  const runnersSig = this.runnersSignature(ctx.baseRunners); // e.g. "1-3"
  const windBucket = Math.round((ctx.weather?.windSpeed ?? 0)/5)*5;

  return [
    ctx.sport, ctx.alertType,
    ctx.inning ?? ctx.quarter ?? 0,
    ctx.outs ?? 0, runnersSig ?? '-',
    ctx.count ? `${ctx.balls}-${ctx.strikes}` : '-',
    scoreBucket, tempoClock, windBucket,
    'v3.1'
  ].join(':');
}
