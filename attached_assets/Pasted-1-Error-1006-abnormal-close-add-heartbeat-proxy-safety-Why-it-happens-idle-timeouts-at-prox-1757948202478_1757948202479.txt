1) Error 1006 (abnormal close) → add heartbeat + proxy safety

Why it happens: idle timeouts at proxies/CDNs, missing ping/pong, TLS mismatch (https page + ws://), or server closing sockets under load.

Server patch (Node + ws):

// ws-setup.ts
import { WebSocketServer, WebSocket } from "ws";
import type { Server } from "http";

export function createWSS(server: Server) {
  const wss = new WebSocketServer({ server, path: "/ws", clientTracking: true });

  // Keep connections alive (<= 30s beats most proxy idle drops)
  const PING_INTERVAL = 25_000;

  wss.on("connection", (ws: WebSocket) => {
    (ws as any).isAlive = true;

    ws.on("pong", () => ((ws as any).isAlive = true));
    ws.on("error", (e) => console.error("WS error:", e));
    ws.on("close", (code, reason) => {
      console.log("WS closed:", code, reason?.toString());
    });
  });

  const interval = setInterval(() => {
    for (const ws of wss.clients) {
      const sock = ws as any;
      if (!sock.isAlive) { ws.terminate(); continue; }
      sock.isAlive = false;
      ws.ping();
    }
  }, PING_INTERVAL);

  wss.on("close", () => clearInterval(interval));
  return wss;
}


HTTP server keepalive (prevents Node from closing idly):

// server.ts
import { createServer } from "http";
const server = createServer(app);
// Disable server timeouts for long-lived WS/SSE

Reverse proxy (Nginx) must pass upgrades:

Client must use wss:// on https pages:

2) this.healthMonitor.startMonitoring is not a function

This means the instance isn’t what you think (wrong import, undefined, or an object missing the method). Make it a tiny, explicit class and ensure a single instance.

// services/health-monitor.ts
import { WebSocketServer } from "ws";

export class HealthMonitor {
  private interval?: NodeJS.Timeout;
  constructor(private wss: WebSocketServer) {}

  startMonitoring() {
    if (this.interval) return; // idempotent
    this.interval = setInterval(() => {
      const metrics = {
        ts: Date.now(),
        clients: this.wss.clients.size,
      };
      // Optionally publish to an admin channel or logs
      // console.log("WS metrics", metrics);
    }, 10_000);
  }

  stopMonitoring() {
    if (this.interval) clearInterval(this.interval);
    this.interval = undefined;
  }
}


Wire it up where you create the WSS:

import { createWSS } from "./ws-setup";
import { HealthMonitor } from "./services/health-monitor";

const wss = createWSS(server);
const healthMonitor = new HealthMonitor(wss);
healthMonitor.startMonitoring();


If you previously exported a plain object, replace it with the class above and ensure you call new HealthMonitor(wss). Also search your codebase for duplicate healthMonitor definitions.

3) Multiple connections / pooling issues

Clients keep opening new sockets if they think the old one died. Ensure a singleton per page and one leader across tabs.

Client singleton + backoff with jitter:

// ws-client.js
let socket;
let attempts = 0;
let explicitlyClosed = false;

export function connect(onMessage) {
  if (socket && socket.readyState === WebSocket.OPEN) return socket;
  const url = `${location.protocol === "https:" ? "wss" : "ws"}://${location.host}/ws`;
  socket = new WebSocket(url);

  socket.addEventListener("open", () => {
    attempts = 0;
    // (optional) identify session
    socket.send(JSON.stringify({ t: "hello", ua: navigator.userAgent }));
  });

  socket.addEventListener("message", (ev) => onMessage?.(ev.data));

  socket.addEventListener("close", () => {
    if (explicitlyClosed) return;
    const delay = Math.min(10_000, 500 * 2 ** attempts) + Math.random() * 500;
    attempts++;
    setTimeout(() => connect(onMessage), delay);
  });

  socket.addEventListener("error", () => socket?.close());
  return socket;
}

export function close() {
  explicitlyClosed = true;
  socket?.close();
}

4) Silent data loss / “hasData:false” even though messages arrive

That’s usually hydrate-then-stream ordering or missing de-duplication/ordering on the client.

Fix pattern:

On load (or reconnect), GET a snapshot /api/alerts/state?user=... and render it.

Only then attach the WS and apply incremental updates (with monotonic seq numbers).

Buffer out-of-order WS messages until the snapshot is loaded and lastSeq is known.

Server: include seq and type in every event

// event payload example
{ type: "ALERT_UPSERT", seq: 10293, alert: {...} }
{ type: "ALERT_DELETE", seq: 10294, id: "..." }


Client: buffer until ready

let ready = false;
let lastSeq = 0;
const buffer = [];

async function hydrate() {
  const res = await fetch("/api/alerts/state");
  const { data, lastSeq: s } = await res.json();
  renderSnapshot(data);
  lastSeq = s;
  ready = true;
  // flush buffered WS events in order
  buffer.sort((a,b) => a.seq - b.seq).forEach(applyEvent);
}

function onWsMessage(raw) {
  const msg = JSON.parse(raw);
  if (!ready) { buffer.push(msg); return; }
  if (msg.seq <= lastSeq) return; // de-dupe
  applyEvent(msg);
  lastSeq = msg.seq;
}

5) Auth 401 on /api/auth/user + WS not inheriting session

Browsers don’t automatically “share” your HTTP session with WS unless:

You’re using cookie-based sessions on the same domain and validating cookies during the upgrade, or

You pass a token (JWT) via Sec-WebSocket-Protocol or a query param and verify it server-side.

Option A: cookie session during upgrade (Express)

// share session middleware with WS
import session from "express-session";
const sessionMiddleware = session({ /* your store + secret */ });
app.use(sessionMiddleware);

import { WebSocketServer } from "ws";
const wss = new WebSocketServer({ noServer: true });

server.on("upgrade", (req, socket, head) => {
  // Run the session middleware on the upgrade request
  sessionMiddleware(req as any, {} as any, () => {
    if (!(req as any).session?.userId) {
      socket.write("HTTP/1.1 401 Unauthorized\r\n\r\n");
      socket.destroy();
      return;
    }
    wss.handleUpgrade(req, socket, head, (ws) => {
      wss.emit("connection", ws, req);
    });
  });
});


Option B: JWT in subprotocol (works great behind CDNs)

// client
const token = await getJwtSomehow();
const ws = new WebSocket(url, ["chirpbot.v1", `auth.${token}`]);

// server
wss.on("connection", (ws, req) => {
  const protos = req.headers["sec-websocket-protocol"]?.split(",").map(s=>s.trim()) || [];
  const authProto = protos.find(p => p.startsWith("auth."));
  const token = authProto?.slice(5);
  const user = verifyJwt(token); // throws on invalid
  (ws as any).userId = user.id;
});


Either approach lets you scope alerts by user on the server and avoids 401s seen by the client later.

6) No graceful degradation → add SSE fallback

If WS fails (1006 loops, corporate networks, strict proxies), SSE gives you push without upgrade gymnastics.

Server SSE endpoint:

// routes/sse.ts
import { Router } from "express";
export const sse = Router();

sse.get("/events", (req, res) => {
  res.setHeader("Content-Type", "text/event-stream");
  res.setHeader("Cache-Control", "no-cache");
  res.setHeader("Connection", "keep-alive");
  res.flushHeaders();

  const send = (event) => res.write(`data: ${JSON.stringify(event)}\n\n`);

  // push initial heartbeat so some proxies keep it open
  const iv = setInterval(() => res.write(":keepalive\n\n"), 20_000);

  // subscribe this HTTP response to your alert bus
  const unsub = alertBus.subscribe((event) => {
    // Optionally include an id: for Last-Event-ID resume
    res.write(`id: ${event.seq}\n`);
    send(event);
  });

  req.on("close", () => { clearInterval(iv); unsub(); });
});


Client fallback helper:

async function startRealtime(onEvent) {
  // 1) Try WS
  try {
    const ws = connect(onEvent); // from earlier snippet
    return;
  } catch {}

  // 2) Fallback to SSE
  try {
    const es = new EventSource("/sse/events", { withCredentials: true });
    es.onmessage = (e) => onEvent(e.data);
    es.onerror = () => es.close();
    return;
  } catch {}

  // 3) Last resort: long-poll
  setInterval(async () => {
    const r = await fetch("/api/alerts/changes?since=" + lastSeq);
    const events = await r.json();
    events.forEach(onEvent);
  }, 3000);
}

7) “New WebSocket connection established (2 total)” spam

This is typically:

Creating sockets inside React renders/effects without proper cleanup.

Hot-reload spinning duplicates in dev.

React effect guard:

useEffect(() => {
  let mounted = true;
  const ws = connect(onWsMessage);
  return () => {
    mounted = false;
    ws?.close();
  };
}, []); // empty deps → create once

8) Mobile resilience (network changes)

Mobile radios flip between Wi-Fi/LTE. Listen for online/offline and force reconnect.

window.addEventListener("online", () => connect(onWsMessage));
window.addEventListener("offline", () => close());
